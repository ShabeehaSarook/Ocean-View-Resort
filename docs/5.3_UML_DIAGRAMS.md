# 5.3 UML Diagram (Class, Sequence, Use Case, and Additional)

This section documents the design of the Ocean View Resort system using UML models derived from the implemented codebase. The diagrams focus on structure, runtime behavior, user interactions, and deployment-level composition.  
All models below are original and aligned to the current project packages, role flows, and database behavior.

## 5.3.1 Class Diagram

The class diagram captures the layered architecture used in this project:

1. Presentation and control classes (`Servlet` controllers and filters) handle HTTP requests, authentication, and route-level authorization.
2. Service classes hold business rules such as reservation status transitions, billing calculation, and registration validation.
3. DAO interfaces define persistence contracts, while JDBC DAO implementations execute SQL through a shared connection utility.
4. Domain model classes represent persistent entities (`User`, `GuestProfile`, `RoomType`, `Room`, `Reservation`, `Bill`).
5. Utility classes provide reusable concerns such as DB connectivity, password hashing, and role-based redirect logic.

The model also shows inheritance (`GuestBaseServlet` -> guest controllers), interface implementation (`*JdbcDAO` -> `*DAO`), and key object associations from the relational schema (for example, bills mapped to reservations and users).

## 5.3.2 Sequence Diagram

The sequence diagram models a realistic operational flow used by staff:

1. Staff submits a reservation form.
2. The reservation controller validates guest, room, dates, and business rules.
3. The system checks overlap conflicts before insert.
4. Staff performs check-in, which updates both reservation and room occupancy state.
5. Staff generates a bill, where billing rules validate financial inputs and compute total.
6. DAO layers persist all state changes to MySQL.

This flow demonstrates the system's transaction-like behavior across controller, service, and DAO boundaries. It also includes alternate paths for overlapping reservations and invalid transitions.

## 5.3.3 Use Case Diagram

The use case model organizes functional scope by role:

1. **Admin** handles user management, room type management, room management, and strategic dashboard/report views.
2. **Staff** performs day-to-day reservation operations, check-in/check-out transitions, and bill lifecycle handling.
3. **Guest** can register, manage only own reservations, view only own bills, and view personal dashboard insights.

Shared use cases such as login, logout, dashboard access, input validation, overlap detection, and access enforcement are shown as reusable included behaviors. This makes role boundaries explicit and supports security verification.

## 5.3.4 Additional UML (Component Diagram)

As an additional model, the component diagram presents runtime composition:

1. Browser UI (JSP pages + JavaScript + Chart.js) initiates requests.
2. Tomcat-hosted filters and controllers coordinate business execution.
3. Service and DAO components process rules and persistence.
4. Utility components centralize cross-cutting logic.
5. MySQL stores all business entities.

This diagram is useful for explaining deployment communication boundaries and dependency direction between front end, web layer, business layer, and data layer.
